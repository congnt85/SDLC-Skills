# Document Standards

All artifacts generated by SDLC skills MUST follow these formatting rules.

---

## Heading Hierarchy

| Level | Usage | Markdown | Style |
|-------|-------|----------|-------|
| H1 `#` | Document title — exactly ONE per document | `# Title` | Bold, largest |
| H2 `##` | Major sections | `## Section` | Bold |
| H3 `###` | Sub-sections | `### Sub-section` | Bold |
| H4 `####` | Detail items within sub-sections | `#### Item` | Bold |

### Rules

- Every document starts with H1 title
- Never skip heading levels (no H1 → H3)
- H1 appears exactly once per document
- Use H2 for the main document structure (sections 1, 2, 3...)
- Use H3 for sub-sections within H2
- Use H4 sparingly — prefer tables or lists over deep nesting

---

## Document Header Block

Every artifact MUST start with a metadata header immediately after H1:

```markdown
# {Document Title}

> **Project**: {Project Name}
> **Version**: {1.0 / 1.1 / 2.0}
> **Date Created**: {YYYY-MM-DD}
> **Last Updated**: {YYYY-MM-DD}
> **Status**: {Draft / In Review / Final}
> **Author**: {Name or "AI-Generated"}
```

### Version Numbering

| Change Type | Version Bump | Example |
|-------------|-------------|---------|
| Initial creation | 1.0 | First draft |
| Minor refinement (wording, formatting) | +0.1 | 1.0 → 1.1 |
| Significant content change (new sections, restructured) | +1.0 | 1.1 → 2.0 |
| Promoted to final | Keep current version | Draft 2.1 → Final 2.1 |

---

## Tables

### Formatting Rules

- Always include header row + separator row
- Align columns for readability
- Use `—` or `[TBD]` for unknown values — never leave cells empty
- Use consistent capitalization within a column

### Standard Table Patterns

**Status fields**: Use `Draft` / `In Review` / `Approved` / `Final`

**Priority fields**: Use `Must Have` / `Should Have` / `Could Have` / `Won't Have`

**Confidence markers** (when applicable):
- ✅ `CONFIRMED` — verified by source or user
- 🔶 `ASSUMED` — inferred, needs confirmation
- ❓ `UNCLEAR` — insufficient information

**Score fields**: Use numeric 1-5 scale with consistent interpretation:
- 1 = Critical risk / Very low
- 2 = High risk / Low
- 3 = Moderate / Acceptable
- 4 = Low risk / Good
- 5 = No risk / Excellent

---

## Lists

- Use `-` for unordered lists (not `*`)
- Use `1.` for ordered lists (sequential numbering)
- Use `- [ ]` for checklists
- Indent nested lists with 2 spaces
- Keep list items parallel in grammatical structure

---

## Cross-References

When referencing other artifacts:
- Use relative path: `See [Charter](../charter/final/charter-final.md)`
- Use section anchors: `See [Risk Register — Technical Risks](#technical-risks)`
- Use item IDs when available: `Related to RISK-003`, `Traces to EPIC-002`

---

## Diagrams

- Use Mermaid syntax for all diagrams
- Wrap in ` ```mermaid ` code blocks
- Include a title in the diagram: `title: Diagram Name`
- Add a text description above the diagram explaining what it shows

---

## File Naming

| Type | Pattern | Example |
|------|---------|---------|
| Draft artifacts | `{topic}-draft.md` | `charter-draft.md` |
| Final artifacts | `{topic}-final.md` | `charter-final.md` |
| Versioned drafts | `{topic}-v{N}.md` | `charter-v2.md` |
| Review reports | `review-report.md` | `review-report.md` |

- Use lowercase with hyphens (kebab-case)
- No spaces in filenames
- No version numbers in final artifacts (version tracked inside the document)
